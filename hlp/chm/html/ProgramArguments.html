Program arguments

MattPlot has the following calling convention:

MattPlot [@] [-h] [["+appended line"]...["+appended line"] [file]] ... [["+appended line"]...["+appended line"] [file]]

MattPlot supports the following arguments:
argument string argument description
-char general option
+"string" appends `string´ to next file
"string" path to a file to be opened

General option arguments

Options arguments are case insensitive.
Option Effect Note
-t stay on top Force the main form in foreground (equivalent to  StayOnTop action)
-x Clear all When open a file, clear all existing plots replacing them with new data
-f Same frame When open a file, put all in the current frame
-h or -? help Ehm, quite useless since not redirecting some strings to calling shell

Appended lines arguments

MattPlot accepts another kind of argument: lines appended at the top of text files that are going to be opened. These arguments are preceded by a `+´ character.
For example, if file.txt is:
 file.txt
1  2  3  4
5  6  7  8
Calling: 
C:\> MattPlot +"a line" +"another one" file.txt
Makes things as if file.txt would be:
 file.txt
a line
another one
1  2  3  4
5  6  7  8
This is useful to pass directives to parser externally, without embeddeding them in the data file.

File path arguments

You can open one or more files passing their path as command line argument: 
C:\> MattPlot data1.txt
C:\> MattPlot D:\acq\plot2.mpz "C:\my data\acq.out"
In the first example the text file data1.dat is parsed and its data imported, in the second the plot file plot2.mpz will be opened and then acq.out imported. Note the double quotes surrounding path.
Remember that you can open files/directories also dragging them in plot area or by  Open action.

A note on spaces in arguments

The string containing the arguments passed by command line shell are decomposed using SPACE and TAB characters as token delimiters, so the line: 
C:\> aaa bbb ccc
is decomposed in three strings, so when you must pass a file path containing spaces, you must enclose it in double quotes: 
C:\> MattPlot E:\my data\data.dat
C:\> ??ERROR: invalid file `E:\my´ and `data\data.dat´

C:\> MattPlot "E:\my data\data.dat"
C:\> Ok
Consider now: 
C:\> MattPlot +#resample [0:2:100) C:\my dat\data.dat
Tells the shell to run a program named MattPlot with the four arguments +#resample, [0:2:100), C:\my and data.dat; this generates several errors, because MattPlot will complain about the invalid/missing files (0:2:100), C:\my, dat\data.dat. You can prevent those misunderstandings enclosing arguments in double quotes; the following examples are both correct: 
C:\> MattPlot +"#resample [0:2:100)" "C:\my dat\data.dat"
C:\> MattPlot "+#resample [0:2:100)" "C:\my dat\data.dat"
Note that, as pointed in directive section, extra spaces are part of the correct syntax; for example the following line is properly decomposed but causes parser to complain about an unknown directive name: 
C:\> MattPlot +#resample[0:2:100) "C:\my dat\data.dat"

Prevent further instances

Prefixing the program arguments string with the special character @ prevents the creation of further MattPlot instances; the arguments passed will be processed by the process already running.

This is useful when user want to add by command line different files obtained subsequently to the same plot.

The instance suppression character @ must strictly be the first character of program arguments.
In the following example the first line creates a new MattPlot instance that processes file1, while the second line causes the previous instance to process file2 without creating another one. 
C:\> MattPlot "C:\file1"
C:\> MattPlot @"C:\file2"
The following example shows that the special character @ must strictly be the first character of arguments string: 
C:\> MattPlot @+"#resample [0:2:100)" "C:\my dat\data.dat"
